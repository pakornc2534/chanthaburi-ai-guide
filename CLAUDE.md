# CLAUDE.md

Guidance for Claude Code agents working in this repo.

## Project: TripChan — เที่ยวจันท์ AI Guide

Bilingual (Thai/English) travel guide app for Chanthaburi province with AI chat, AI trip planner, and a check-in / rewards POC.

## Stack

- **TanStack Start** (React 19 + Router, file-based routing) on **Vite 7** + TypeScript
- **Tailwind CSS 4** + **shadcn/ui** components (Radix-based)
- **Supabase** (Postgres + Auth + RLS) — single shared project for the team
- **OpenRouter** for the LLM (default model `google/gemini-2.0-flash-lite-001`, override via `OPENROUTER_MODEL`)
- **Build target:** Cloudflare Workers by default (via `@lovable.dev/vite-tanstack-config` + `@cloudflare/vite-plugin`). For Ubuntu/Docker deploy see `DEPLOY.md` — the target must be switched to Node SSR.

## Conventions

- **Imports:** `@/...` is `./src/...` (path alias).
- **Routing is file-based.** Adding a new route = a new file in `src/routes/`. The generator updates `routeTree.gen.ts` — do not edit that file by hand.
- **Server vs client boundary:**
  - Anything in `src/server-fns/*.functions.ts` and `src/routes/api/*.ts` runs on the server.
  - `client.server.ts` (admin Supabase client) and `process.env.*_SECRET` keys are **server-only**. Never import them from a component or any file that ships to the browser.
  - Browser-side Supabase usage goes through `src/integrations/supabase/client.ts` with the publishable key only.
- **shadcn/ui** lives in `src/components/ui/`. Prefer composing existing primitives over reaching for new Radix packages directly.
- **Tailwind v4** — utility-first; no `tailwind.config.ts`, theming lives in `styles.css` via `@theme`.
- **i18n** — strings live in `src/lib/i18n.tsx`. Use `useT()` for new copy; don't hardcode TH/EN strings in components.
- **State** — TanStack Query for server data; lightweight POC state (points, check-ins) in `localStorage` via `lib/local-state.ts`.
- **Forms** — `react-hook-form` + `zod` resolvers (see existing forms before rolling your own).

## AI / Server functions

- **Chat (streaming):** `POST /api/chat` → forwards SSE from OpenRouter. The server prompt **only allows recommendations from the Supabase `places` catalog** — preserve this constraint when editing.
- **Trip planner:** `generateTripPlan` in `src/server-fns/ai.functions.ts` uses structured output. If you change the schema, also update the consuming component in `routes/planner.tsx`.
- **Model override:** `OPENROUTER_MODEL` env var. Don't hardcode model IDs in new code — read from env with a sane default.
- **Costs:** OpenRouter is metered. Avoid wide-open endpoints; the existing `/api/chat` accepts arbitrary user input and is a candidate for rate-limiting before public deploy.

## Database / migrations

- Schema is in `supabase/migrations/`. Add new migrations as **new files** with a fresh timestamp prefix — never edit applied migrations.
- Tables to know: `places`, `favorites`, `trips`, `partners`, `rewards`. RLS is on; the publishable client cannot bypass it.
- After schema changes, regenerate `src/integrations/supabase/types.ts` (`supabase gen types typescript`).
- Seed data lives in `20260501060000_reseed_places_8_categories.sql` (49 places, 8 categories).

## Env / secrets

- `.env.example` is canonical for the variable list.
- **Never** commit `.env`. It is in `.gitignore`.
- `VITE_*` vars are baked into the client bundle at build time → only put publishable values here.
- `SUPABASE_SERVICE_ROLE_KEY` and `OPENROUTER_API_KEY` are runtime server-only.

## Things to avoid

- Editing `routeTree.gen.ts` by hand (regenerated on every dev/build).
- Importing from `client.server.ts` in any component or client-side module.
- Adding new strings as raw literals — go through `lib/i18n.tsx`.
- Putting a service role key behind a `VITE_*` variable.
- Hardcoding `localhost:8080` or `localhost:5173` in code that ships to prod (use env vars).
- Re-introducing `@lovable.dev/vite-tanstack-config` if it has been removed for a Node deploy — see `DEPLOY.md`.

## Deployment

- **Cloudflare Workers** (default) — `wrangler deploy` with `wrangler.jsonc` after `npm run build`.
- **Ubuntu + Docker + Nginx** at `trip.ipongs.com` — see `DEPLOY.md` for the full runbook including the build-target switch and TLS setup.

## Testing UI changes

There is no automated UI test suite. For visible changes, run `npm run dev`, exercise the route in a browser, and check both `lang=th` and `lang=en` if the change touches user-facing copy. Streaming/AI changes need a real `OPENROUTER_API_KEY` set in `.env`.
